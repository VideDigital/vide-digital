## O que é este projeto

`VideDigital/vide-digital` — SaaS multi-tenant "Vide Hub" em Firebase
(Firestore + Storage + Auth, plano **Blaze**). A maior parte da escrita é
direta do cliente protegida por Firestore Rules; só duas Cloud Functions
vivas em produção hoje: `askBusinessAI` (chat de IA do dono) e
`askPublicBusinessAI` (chat de IA pro visitante da loja pública), as duas
publicadas e confirmadas com resposta real do Gemini. Ver
`docs/FIREBASE_SPARK_ARCHITECTURE.md` pro contrato completo dessa decisão de
arquitetura.

**Convenção deste projeto**: os agentes trabalham direto na branch `main`
(não em PRs) — `main` é a branch de produção real. GitHub Pages faz deploy
automático do frontend a cada push em `main`. As Firestore Rules só vão pra
produção via workflow manual **"Deploy Firebase Spark"**
(`.github/workflows/firebase-deploy.yml`, exige `project_id:
vide-digital-saas` + `confirm_production: DEPLOY`). Cloud Functions vão por
`.github/workflows/firebase-deploy-functions.yml`, também manual.

Documento-mestre: **`docs/ROADMAP_RD3_STATUS.md`** — sempre ler primeiro.
Handoff anterior a este: `docs/HANDOFF_CHATGPT_2026-07-22.md` (contexto de
tudo que veio antes desta sessão — CRM 360, Central de Atendimento,

## O que aconteceu nesta sessão (2026-07-23)

O pedido do usuário no início da sessão foi simples: "qual é a próxima

### Bugs reais encontrados e corrigidos nesta sessão (ordem cronológica)

1. **`firebase-init.js` — `projectId` de produção vazava pro modo Emulator.**
   `connectFirestoreEmulator`/`connectAuthEmulator` só redirecionam o HOST
   da conexão, nunca o `projectId` gravado em `initializeApp()` — como o
   Emulator serve múltiplos "projetos falsos" isolados por `projectId` na
   mesma porta, um `projectId` de produção hardcoded fazia todo login real
   ler/escrever num namespace vazio no Emulator. Fix: `projectId` agora é
   condicional (`usingEmulators ? "demo-vide-hub" : "vide-digital-saas"`),
   com a checagem de `usingEmulators` reordenada pra ANTES do
   `initializeApp()`. Commit `434cb8b`.
2. **`ativarAba(targetId)` em `dashboard-app.js` nunca retornava `true` no
   sucesso** (só tinha `return false` nos caminhos de falha; sucesso caía no
   `undefined` implícito). Fix: `return true;` explícito adicionado antes do
   fechamento da função. Commit `9902bb0`.
3. **Teste corria antes do `VideHubContext` terminar de inicializar** —
   `VideHubContext.initialize()` é assíncrono, chamado dentro do callback do
   `onAuthStateChanged`; o teste chamava `ativarAba()` cedo demais e caía no
   bloqueio de "carregando permissões" mesmo pra quem tinha acesso de
   verdade. Fix: `window.__videHubContextInitialized()` exposto (redundante
   com `VideHubContext.getSnapshot().initialized`, que já era público, mas
   inofensivo) + `page.waitForFunction(...)` no teste antes de qualquer
   `ativarAba`. Commit `9902bb0`.
4. **`tests/emulator/ui/profiles.smoke.mjs` testava uma view que não
   existe** (`view-produtos` — catálogo vive dentro de `view-dashboard`, sem
   view própria; `PERMISSOES_NAV["view-produtos"]` em `dashboard-app.js` é
   entrada morta). Fix: teste trocado pra `view-avaliacoes` (view real,
   mesma permissão `"produtos"`). Commit `566d694`.
5. **Mesmo teste tinha a tabela de permissões esperadas do perfil
   `reader` errada** — assumia que reader = editor menos "editar", mas o
   seed (`scripts/seed-emulator.mjs`) dá ao reader um conjunto de `ver`
   genuinamente menor (sem `"central-ia"` nem `"funcionarios"`). Fix: tabela
   `esperado` corrigida pra bater com o seed real. Commit `566d694`.
6. **`notificacaoVisivelPara()` em `firestore.rules` sem guard antes de
   acessar campo opcional** — faltava `.keys().hasAny([...])` antes de ler
   `dados.uid` num ramo do `or()`, causando erro de avaliação em vez de
   negação limpa para documentos sem esse campo. Fix: guard adicionado,
   espelhando o padrão já usado nos outros ramos. Commit `566d694`.
7. **`buscarNotificacoesRemoto()` em `dashboard-app.js` — query composta
   com `or()` não é "provável" pelo motor de regras do Firestore.** Pra
   `list()`/queries de coleção, o Firestore precisa PROVAR que a regra vale
   pra QUALQUER documento que possa bater com os filtros da query, não só os
   retornados de fato. Uma query com `or()` combinando `where()` em campos
   diferentes torna essa prova intratável quando a regra tem múltiplos
   ramos OR dependendo de campos diferentes — Firestore lança "evaluation
   error" (mas o `err.code` real entregue ao cliente é `permission-denied`
   — a mensagem verbosa com "evaluation error" é ruído de rastro de prova
   interno, não necessariamente um bug real; ver lição #2 abaixo). Fix:
   query composta trocada por 3 (depois 2, ver próximo item) `getDocs()`
   simples e independentes, rodadas via `Promise.all`, deduplicadas por id
   no cliente — cada uma bate exatamente com um ramo da regra. Commits
   `f384dd2`, depois refinado.
8. **Ramo morto no `or()` de notificações** — a terceira query
   (`where("uid", "==", usuarioUID)`) foi removida: nenhum código do
   projeto grava um campo `uid` bruto em `notificacoes` (`admin.html` só
   usa `destinatarios`, seja a string `"todos"` ou uma lista de uids).
   Commit `ca83b20`.
9. **`renderizarPrimeirosPassos()` e `atualizarKpisDashboard()` em
   `dashboard-app.js` — permission-denied esperado tratado como falha
   real.** Funcionário sem a permissão `"landing-pages"` recebe
   `permission-denied` (correto e esperado) ao consultar `landing_pages`,
   mas a ausência de `.catch()` derrubava a Promise inteira e escondia o
   checklist/KPIs COMPLETOS — inclusive as partes que o funcionário TEM
   acesso. Fix: `.catch(() => null)` + guards de null. Commit `ca83b20`.
   (Um bug secundário nesse mesmo fix — `lpSnap.forEach` sem guard de null
   depois do `.catch()` — foi pego e corrigido antes mesmo de chegar no
   CI, revisando o próprio diff.)
10. **`carregarCampanha()` em `dashboard-app.js` — mesmo padrão do item 9,
    mas manifestado como poluição de console em vez de tela quebrada.** O
    `catch(err) { console.error(err); }` sempre logava, inclusive pra
    `permission-denied` esperado (funcionário sem `"campanhas"`) — e a
    suíte de teste capta `console.error` como falha de JS
    (`coletarErrosConsole` em `_helpers.mjs`). Fix: só loga se
    `err?.code !== "permission-denied"`. Commit `ca83b20`.
11. **Bug de Rules mais sutil da sessão: funcionário não conseguia ler
    NENHUMA notificação da própria loja.** `notificacaoVisivelPara()`
    comparava `request.auth.uid in dados.destinatarios` — mas
    `dashboard-app.js` sempre consulta notificações usando `usuarioUID =
    contextoVide.effectiveUid || user.uid` (o uid do DONO da loja, não da
    conta do funcionário — `admin.html` grava `destinatarios` como uids de
    LOJA/dono, nunca de funcionário individual). Pra owner, `effectiveUid
    === request.auth.uid`, então funcionava por coincidência. Pra
    funcionário, o valor do filtro `array-contains` da query (uid do dono)
    NUNCA bate com `request.auth.uid` (uid do funcionário) — o Firestore não
    consegue provar nenhum ramo da regra, evaluation error. **Diagnosticado
    e corrigido com teste local usando `@firebase/rules-unit-testing`**
    (script descartável, não commitado — ver seção "Como testar Rules
    localmente sem gastar rodada de CI" abaixo) ANTES de gastar mais uma
    rodada de CI de ~15min, incluindo um teste negativo confirmando que
    funcionário de OUTRA loja continua sem acesso. Fix: ramo extra
    `employeeExists() && employeeDoc().data.donoUID in dados.destinatarios`
    na função. Commit `8ff3575`.
12. **Bug no PRÓPRIO TESTE, não no app**: `tests/emulator/ui/
    pedidos.flow.mjs` tentava `page.fill("#ped-prazo-entrega", "5 dias
    úteis (QA)")` — mas esse campo é genuinamente `<input type="date">` no
    `dashboard.html` (alimenta `{{prazo_entrega}}` nos templates de
    atendimento), e Playwright recusa `fill()` com texto livre em input de
    data ("Malformed value"). Fix: valor trocado pro formato ISO
    (`YYYY-MM-DD`) que o input realmente aceita. Commit `af4ad5b`.

### Lição importante #1 — como diagnosticar sem gastar rodadas de CI

Ao investigar os bugs #7 e #11 (ambos envolvendo "evaluation error" do
Firestore), a primeira suspeita foi que HAVIA mais bugs de Rules em
`campanhas`/`landing_pages` (mensagens de erro pareciam idênticas). Em vez
de aplicar um fix especulativo e gastar outra rodada de ~15min de CI pra
descobrir se funcionou, foi usado **`@firebase/rules-unit-testing`
diretamente** (o mesmo pacote que já sustenta `tests/emulator/
firestore-security.test.mjs`), com scripts descartáveis criados em
`tests/emulator/scratch-*.test.mjs` (sempre DELETADOS depois de confirmar o
diagnóstico, nunca commitados):

```js
import { initializeTestEnvironment } from "@firebase/rules-unit-testing";
const testEnv = await initializeTestEnvironment({
  projectId: "demo-vide-hub",
  firestore: { rules: fs.readFileSync("firestore.rules", "utf8") }
});
await testEnv.withSecurityRulesDisabled(async (context) => {
  // seed direto, sem passar pelas Rules
});
const db = testEnv.authenticatedContext("algumUid").firestore();
// roda a query/get exata que está falhando, inspeciona err.code
await testEnv.cleanup();
```

Rodar com: `pnpm exec firebase emulators:exec --project demo-vide-hub --only
firestore "node --test caminho/do/scratch.test.mjs"` (pode passar um
`firestore.rules` alternativo via variável de ambiente customizada pra
comparar "antes" e "depois" da regra num único script, sem tocar no arquivo
real até confirmar).

Isso evitou pelo menos duas correções desnecessárias/erradas: o problema em
`campanhas`/`landing_pages` acabou sendo só tratamento de erro no
`dashboard-app.js` (item 9), NÃO um bug de Rules — `err.code` já era
`permission-denied` corretamente, a mensagem verbosa com "evaluation error"
era só ruído de rastro interno de prova. Já o problema de notificações (item
11) era mesmo um bug de Rules — e o mesmo harness local confirmou o fix
exato antes de gastar CI.

**Regra prática**: quando aparecer "evaluation error" nos logs, sempre
confirmar `err.code` antes de assumir que é um bug de Rules — pode ser uma
negação correta com mensagem verbosa.

### Lição importante #2 — por que `or()` composto em query de list() quebra

Firestore precisa PROVAR estaticamente, pra qualquer `list()`/query de
coleção, que a regra de segurança vale pra TODO documento que possa bater
com os filtros — não só os retornados de fato. Uma regra com múltiplos
ramos `or()` dependendo de campos diferentes, combinada com uma query
também usando `or()` de filtros em campos diferentes, geralmente é
intratável pra esse provador estático → "evaluation error". **Padrão que
funciona**: várias queries SIMPLES e independentes (um filtro cada),
mescladas no cliente (dedupe por id), onde cada query bate exatamente com
UM ramo da regra. Visto duas vezes nesta sessão (`buscarNotificacoesRemoto`)
e já documentado antes pra `collectionGroup` em
`docs/HISTORICO_EVENTOS_ATENDIMENTO.md` (Fase 19) — é o mesmo tipo de
limitação aparecendo de formas diferentes.

## Padrões e armadilhas gerais do repo (herdados do handoff anterior, ainda válidos)

- **`collectionGroup` + Rules com `get()`/`exists()` + catch-all `match
  /{document=**} { allow read, write: if false; }`**: historicamente não
  funciona neste ruleset (estoura teto de 1000 expressões). Ver Fase 19 de
  `docs/HISTORICO_EVENTOS_ATENDIMENTO.md`.
- **Query composta com `or()` em `list()`**: ver "Lição importante #2"
  acima — é o mesmo tipo de limitação do item anterior, achado de novo
  nesta sessão em contexto diferente (notificações).
- **CSS de layout com `min-height` sem `height`**: em grid/flex aninhado
  com `overflow-y: auto` num filho, `min-height` sozinho não limita o
  container.
- **Padrão de módulo**: cada feature grande vira um arquivo próprio
  (`crm360.js`, `atendimento.js`, `templates-atendimento.js`,
  `pedidos-estruturados.js`, `base-conhecimento-ia.js`, `ia-copilot.js`)
  exportando funções puras testáveis + `criarXController(deps)` por
  injeção de dependência (nunca importa Firebase direto, exceto
  `core/vide-context.js`).
- **`usuarioUID` em `dashboard-app.js` é `effectiveUid`, não
  `auth.uid`**: pra funcionários, sempre é o uid do DONO da loja (tenant),
  não da conta pessoal do funcionário. Qualquer regra nova em
  `firestore.rules` que precise comparar contra o usuário logado
  PRECISA considerar esse caso — comparar só `request.auth.uid` funciona
  pro dono por coincidência, mas nunca pra funcionário (foi exatamente o
  bug do item 11 acima).
